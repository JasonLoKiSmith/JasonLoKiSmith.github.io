3. Relational Databases (SQL)

Microsoft SQL and MySql mostly, although I love the JSON features from Postgres!

Relational Databases (SQL)

Microsoft SQL (MSSQL)

I have been working with Microsoft SQL for probably about 90% of my professional career. So I will likely be able to tell you a lot more of what I have done in this realm. This would be things like query optimization, checking clustered versus non-clustered indexes to improve database query performance, deploying DACPAC files, tweaking isolation levels when running certain operations where a dirty read is considered "ok", to creating CLR procedures to run some CLR code a little closer the database server.

Document Storage

For one of my personal projects, the client had a requirement that required my database to house hundreds and eventually thousands of documents, I knew storing lots of files in the database would be detrimental to performance, and the fact that the website would have to run on SQL Express, which has a 10 GB file limit for both the MDF and the LDF combined. Luckily though the hosting plan was to host it on a Virtual Private Server (VPS), which meant that I had control over what version of SQL would be running there. That was when I discovered the FileStreams feature in SQL 2017, which really saved me. I quickly realized the benefits of using FileStreams.
Benefits of FileStreams

  • Data is stored on the file system directly which means increased I\O performance
  • Files can easily be stored across multiple storage units, again, increasing I\O. (Documents and system critical tables do not have to be on the same hard-drive, you can split them up across multiple hard-drives, which means increased performance in parallel I\O operations)

MySql

I worked with MySql for about a year and a half back in 2010 up until mid 2011. I worked on some software for a big investment firm which ran on MySql. I grew to dislike MySql very quickly, but to be fair, this was likely due to the specific version of MySQL I was forced to work on. This specific version did not support stored procedures, functions, and the only datatype you could cast to is CHAR. This was frustrating to me as I had to adopt bad practices in order to develop on MySql.

4. Back-end Services

Windows Services, SOAP, RESTful WebAPI, CRON Jobs etc.

Back-end Services

Windows Services

Windows services was once upon a time the only way to properly run back-end jobs, I have written numerous multi-threaded back-end services using timers, purely because of timers running on their own threads by design. This made multi-threading a lot easier back in the days of .NET 3.5. Times have changed a lot since then, and these days my go to library for windows services would probably be Quartz.NET and Topshelf

Quartz (CRON) & Topshelf

Quartz is a very common library amongst developers, so I wont elaborate on it, however not every developer knows of Topshelf's existence. Topshelf makes the one pain about writing windows services so much easier - debugging. It offers seamless differentiation between running as a console application (Visual Studio) and running as an actual windows service! So yes I would recommend Topshelf to any .NET developer!

SOAP and RESTful

I have created WCF SOAP services and worked with RESTfull services in both .NET Framework and .NET Core (2 & 3) quite extensively. To be fair it has been more in .NET Core 2 than 3, purely because at the time of writing this, Core 3 was just released.
